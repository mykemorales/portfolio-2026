Problem · Service Offerings

In the Discovery leading up to the MVP launch, I had identified a critical gap: the old process could only accommodate one service per Jobsheet.

An estimated 40% of incoming requests required multiple services, which meant without a solution, intake colleagues would be forced to fragment requests across multiple Jobsheets and piece the relationship together manually, at a scale of around 12,000 requests per week.

Approach

Auditing the existing service catalog revealed an important consideration: additional media types were on the roadmap, meaning a component built solely around PowerPoint decks would need rebuilding sooner than later.

I proposed a phased approach to keep delivery on track while avoiding a near-term rebuild. With the business requirements clarified, the logic was straightforward: one media type per request, with the media offer dropdown updating dynamically based on the type selected.

Solution

  • Phased component architecture to accommodate future media types without delaying the sprint
  • Media type selector positioned above the offer table to establish visual hierarchy and dictate the offers below
  • String input field for slide numbers to handle free-form, variable slide selections across any deck size
  • Inline PP total as a self-initiated addition to reduce manual calculation and support work distribution
  • Validation tradeoff, adapting to ServiceNow out-of-box error behaviour rather than the original inline approach
  • Known defect shipped with a logged remediation plan to fix the disabled look of the default state

The running PP total was a self-initiated addition that removed a consistent source of mental load. Previously, intake colleagues calculated cumulative production points manually across multiple offers, a task prone to error when dividing work across production colleagues. Automating it was well received by both intake colleagues and team leads following launch.

Impact

The service offering component contributed to a broader set of platform improvements that collectively reduced average request processing time from 8:38 to 7:32 minutes across 130,000 monthly requests, saving an estimated 2,300+ hours of intake capacity per month.

Most directly, the component eliminated the fragmented Jobsheet workaround that had been affecting an estimated 12,000 requests every week. Intake colleagues could now define all service requirements within a single Jobsheet, removing the manual overhead of splitting, tracking, and reconnecting related requests.

The pinned comment feature was the single most requested addition from intake colleagues across the platform. Prior to its introduction, instructions added by intake colleagues were buried within activity streams containing several dozen tracked items, requiring both intake and production colleagues to manually scroll through to locate them on every request.

With instructions now pinned and immediately visible, an estimated 20 seconds of searching per user per request translates to over 1,400 hours of collective time saved across the platform every month.
